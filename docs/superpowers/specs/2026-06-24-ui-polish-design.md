# UI/UX 打磨增量 —— 设计 (Drag Timeline + 战斗视觉 + 全中文 + 招式图鉴)

- 日期: 2026-06-24
- 状态: 已批准设计,待拆实现计划
- 基线: 战斗垂直切片已合并 master(43 测试绿)。本增量在其上做表现层打磨。
- 引擎: Godot 4.6.1

## 1. 目标与范围

把已能玩的切片在**操作手感、战斗视觉、全中文、信息可查性**四方面打磨好,**不扩展游戏内容**。
这是一次表现层(`src/scenes/*`)为主的增量;**战斗模拟器逻辑不改**(所有需要的事件已在产出)。

### 1.1 范围【包含】

1. **排招:拖拽 + 自动吸附的时间轴**(替换现在的「选中招式→点格子」)。
2. **观战:视觉打磨**——绿色血条 + 红色掉血段消退动画、血量/体力数字、每拍特殊事件的中文浮动文字。
3. **全中文**——对手意图、事件日志、拍数标签。
4. **招式图鉴 / 出招表面板**——可开关,列出每张招式的完整说明 + 所有连招配方。

### 1.2 范围【不包含】(后续各自立项)

更多招式/流派、更多敌人/难度、新机制词缀、爬塔 run 结构、美术与音效、存档、Steam。

### 1.3 成功标准

- 玩家用拖拽即可排招,空拍即停顿,吸附顺手;非法摆放被拒绝。
- 观战时血条掉血可见、数字清晰、关键事件有中文提示。
- 界面无英文(招名/事件/意图/拍数均中文)。
- 图鉴面板能查到每张招式数据与全部连招配方。
- 可测的逻辑(中文意图、吸附/校验、图鉴数据装配)有 GUT 覆盖;动画/拖放交互由实例化测试 + 用户手动试玩验证。

## 2. 排招:拖拽 + 吸附时间轴

### 2.1 交互

- **时间轴**为 `N_TICKS` 个等宽拍格(横向)。`N_TICKS` 从 10 **提高到 14**(`fight.gd` 与 `plan_phase` 同步)。
- 手牌区每张招式是一个**可拖动源**(Godot `Control._get_drag_data`)。
- 把招式**拖到时间轴**并松手:**吸附**到光标所在拍(并在紧邻已排招式边缘时吸附贴合)。落点即该招式的 `start` 拍。
- 招式之间**空着的拍 = 停顿**(无需显式停顿招式)。
- 已排招式渲染为**横跨其 `total_duration()` 拍的名字块**:
  - **拖动名字块**可改其 `start`(同样吸附、同样校验)。
  - **点击名字块(或拖出时间轴)= 取消**该招式,空出其占用拍(成为停顿;**不**自动贴左,因为间隔是玩家有意留的)。
- **落点实时校验**(用现有 `Plan.is_valid(sta_max, n_ticks)` + 重叠检测):不能与其他招式重叠、必须在 `[0, N_TICKS)` 内且整段放得下、所放后总体力 ≤ `floor(1.5 * sta_max)`。非法则**吸附失败**(招式弹回/落点高亮红色),`_plan` 不变。
- 顶部仍显示:**体力** `已排/上限 (可超额至 1.5x)`、**连招预览**(`rules.apply` 后的名字)、**对手意图**(中文)。

### 2.2 实现要点

- `plan_phase.gd` 重写放置逻辑:用 Godot 拖放(`_get_drag_data`/`_can_drop_data`/`_drop_data`)或在时间轴容器上手动处理 `gui_input` 的按下/拖动/松开。落点拍 = `floori(local_x / tick_width)`,再做吸附(就近 + 贴邻)。
- 内部仍维护一个 `Plan`;每次放置/移动/删除后 `_refresh()` 重算体力标签、连招预览,并重绘时间轴(占用块 + 空拍)。
- 时间轴重绘:为每个已排 `PlacedMove` 画一个跨 `total_duration()` 拍、显示 `move_name` 的块;空拍画灰底。
- 提交(`_on_commit`)逻辑不变:`plan_committed.emit(_rules.apply(_plan))`。

### 2.3 可测的纯逻辑(从 UI 中抽出以便单测)

把「落点 → 是否可放 → 放置后的 Plan」抽成不依赖节点的纯函数,例如:
- `func snap_tick(local_x: float, tick_width: float) -> int`
- `func can_place(plan: Plan, move: Move, start: int) -> bool`(重叠 + 边界 + 1.5x 体力)
- `func with_move(plan: Plan, move: Move, start: int) -> Plan` / `func without_index(plan, idx) -> Plan`
这些可在 `test_plan_phase.gd` 里直接断言;拖放手势本身由实例化测试覆盖(节点存在、setup 后时间轴/手牌数量正确)。

## 3. 观战:视觉打磨

仍是 `watch_phase`,逐拍回放事件日志,增加:

- **血条**:底色绿色。受到伤害(`hit`/`interrupt`/`throw_break`,`amount>0`)时,被扣的一段先以**红色**显示,短暂(约 0.4s)后**消退**至当前血量(用一个「红色追赶条」或 Tween:前景绿条立即降到新值,背景红条延迟补间到新值)。
- **数字标注**:每条血条/体力条上叠加 `Label` 显示 `当前/上限`(如 `34/40`、`体力 6/10`),随事件更新。
- **浮动文字**:每拍若事件触发,在对应角色一侧弹出**中文浮动 `Label`**(短暂上浮淡出)。映射:
  - `hit` → `命中 -N`(N 为伤害);若目标喘息中加 `(破绽)`
  - `interrupt` → `打断!`
  - `throw_break` → `投破防! -N`
  - `block` → `格挡` (若该拍伴随防御方 `stamina` 正值 → `格挡 +K体力`)
  - `whiff` → `用力过猛`(攻方)/ `闪避!`(防方一侧,见下)
  - `exhaust` → `气力不继!`
  - 连招命中(`move_id` 属于连招结果如 `chain_kick/wuying/qiankun`)→ 额外弹该招中文名(`连环踢!`等)
  - `death` → `败北` / 由 `fight` 的结果标签处理
- **闪避提示**:`whiff` 事件意味着攻击被闪避(攻方用力过猛)。在**防御方**一侧弹 `闪避!`,在攻方一侧弹 `用力过猛`。(`whiff` 事件已含攻方 `actor`,防方为 `1-actor`。)

实现:`_apply_event` 扩展为：更新血/体力条(含红色追赶动画)、更新数字 Label、生成浮动文字节点(自管理上浮+淡出+`queue_free`)。事件→中文文案集中在一个 `_event_text(e) -> String` / 映射字典里,便于改文案。

## 4. 全中文

- **对手意图**:`AiPlanner.intent` 当前返回招式 **id**(英文)。改为返回**中文显示名**:已揭示的槽位用 `move.move_name`,未揭示用 `"？"`(全角问号)。返回类型相应改为 `Array[String]`。`plan_phase` 的意图标签直接拼接这些中文名。
  - 注意:敌方计划已 `rules.apply` 融合,意图里可能出现连招结果名(如 `连环踢`),这是期望行为(看出对方在蓄大招)。
- **事件日志 / 拍数**:`watch_phase` 的日志行改为中文事件文案(复用 §3 的 `_event_text`),拍数标签 `tick N` → `第N拍`。
- 既有中文(手牌名、连招预览、胜负 `胜利!/败北...`)保持。

## 5. 招式图鉴 / 出招表面板

- 新增 `src/scenes/codex.{gd,tscn}`(一个可显示/隐藏的 `Control` 覆盖面板),由 `fight.tscn` 里一个「图鉴」按钮开关(`PlanPhase` 期间可用,不影响对局状态)。
- **招式列表**:遍历 `Deck.starter()` + 连招结果(`Deck.chain_kick()/wuying()/qiankun()`),每项显示:中文名、类型(攻/格挡/闪避/投)、体力消耗、伤害、前摇/命中/后摇、词缀(打断/霸体/重击)、标签。
- **出招表**:从 `ComboLibrary.build()` 的配方读出并以中文呈现(`腿法+腿法+腿法 → 连环踢`、`攻+防+投 → 乾坤大挪移` 等)。需要 `ComboRules` 暴露只读的配方列表 + 槽位的人类可读描述(新增 `ComboRules.describe_recipes() -> Array[Dictionary]` 或公开 `_recipes`)。
- 类型/标签/词缀的中文映射集中在一处常量表,供图鉴与浮动文字共用。

## 6. 架构与影响面

- 改动集中在 `src/scenes/plan_phase.*`、`src/scenes/watch_phase.*`、`src/scenes/fight.*`、新增 `src/scenes/codex.*`,以及 `src/ai/ai_planner.gd`(intent 改中文)。
- `src/combat/` **仅一处**轻微改动:`combo_rules.gd` 暴露配方供图鉴读取(只读访问,不改匹配逻辑)。模拟器结算逻辑不动。
- `N_TICKS` 常量 10 → 14(`fight.gd` 与 `plan_phase` 默认值)。
- 共享中文映射(事件文案、类型/标签/词缀名)放一个小工具(如 `src/scenes/labels.gd` 或 `src/content/loc.gd`)避免重复。

## 7. 测试策略

- **纯逻辑(GUT 单测)**:`AiPlanner.intent` 返回中文名 + 全角问号;`snap_tick`/`can_place`/`with_move`/`without_index` 的吸附与校验;`ComboRules.describe_recipes` 的内容;中文映射表覆盖所有事件类型/招式类型。
- **场景实例化测试**:`plan_phase`/`watch_phase`/`codex`/`fight` 实例化、节点接线、setup 后数量正确(沿用切片已有模式)。拖放手势可用程序方式调用放置纯函数 + 直接调 `_drop_data` 等价入口验证。
- **动画与拖拽交互**:由用户手动试玩验证(本机不可视化时,实例化测试 + 纯逻辑测试兜底)。
- 全套 `bash run_tests.sh` 必须保持绿。
